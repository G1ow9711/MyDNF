# 主线 NPC 对话与任务交付设计

## 目标

补齐当前任务系统缺少的剧情呈现。玩家在任务面板查看进行中任务时可打开简短任务简报；目标达成后必须经过 NPC 交付对话才能领取奖励。对话保持短、可跳过、可键盘推进，并沿用现有任务触发、奖励和解锁规则。

## 权威行为

- 对话是 `AppModel` 运行时状态，不进入版本化本地存档。
- `active` 任务打开 `briefing`，结束后回到任务面板，不改变任务状态或奖励。
- `ready` 任务打开 `turn-in`，只有最后一段推进或主动跳过时调用现有 `claimQuestReward()`。
- 已完成或锁定任务不能启动对话；非法、过期或重复交付不得重复领取奖励。
- 任务奖励和后续任务激活仍由 `claimQuestReward()` 负责，避免第二套结算逻辑。

## 内容结构

- 新建数据驱动的任务对话目录，每条包含任务、阶段、说话人、NPC 身份、短台词和肖像位置。
- 首版覆盖全部七个现有任务，使用五名城镇角色：公会司书、炉山铁匠、获救商人、琉璃鉴定师、天衣掌柜。
- 每个阶段保持 2-4 句，不放长文本墙。
- 对话画面使用炉山市集环境、人物肖像、章节/任务名、说话人、进度和明确操作。

## 控制与界面

- 任务列表分别显示“查看简报”或“交付任务”。
- 对话模式隐藏顶部系统导航，防止误切换。
- `Enter` / `Space` 推进；`Escape` 或“跳过”结束。
- `turn-in` 最后一段和跳过都会合法结算一次；`briefing` 只关闭。
- 挂载稳定的 `data-story-dialogue`、任务、阶段、步骤、NPC 和按钮状态钩子。

## 验收

1. 领域/应用测试证明简报不改变存档，交付只在结尾结算一次，过期对话不能重复领奖。
2. UI 测试覆盖剧情画面、肖像、步骤、推进/跳过按钮和隐藏导航。
3. 真实浏览器从 `ready` 主线进入对话，使用真实 Enter 推进，确认奖励、任务解锁和 `localStorage` 同步。
4. 真实浏览器验证进行中任务用 Escape 跳过后没有奖励或状态变化。
